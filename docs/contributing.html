Contributing to BRAPHIN

Thank you for your interest in contributing! This page explains how to report bugs, propose features, set up a development environment, and submit code.

On this page

Academic integrity: BRAPHIN is a scientific library. Do not submit plagiarised code or fabricated benchmark results. All contributors are expected to be respectful and constructive.

Reporting Bugs

Before opening an issue, please:

  1. Search existing issues to avoid duplicates.
  2. Reproduce the bug against the latest main branch.
  3. Include a minimal reproducible example — synthetic or anonymised data only, never real patient data.

A good bug report contains:

  • BRAPHIN version: import braphin; print(braphin.__version__)
  • Python version and operating system
  • Full traceback
  • Minimal code snippet that reproduces the issue

Requesting Features

Open a GitHub issue with the enhancement label. Describe:

  • The neuroimaging use case that motivates the feature
  • Which pipeline stage it would affect
  • Any relevant references (papers, toolbox implementations)

Features already listed in the Unreleased section are planned — contributions implementing them are especially welcome.

Development Environment

# 1. Fork and clone
git clone https://github.com/<your-username>/braphin.git
cd braphin

# 2. Create a virtual environment
python -m venv .venv
source .venv/bin/activate     # Windows: .venv\Scripts\activate

# 3. Install in editable mode with dev dependencies
pip install -e ".[dev]"
# — or —
pip install -r requirements-dev.txt && pip install -e .

# 4. Optional: EEG or GNN extras
pip install -e ".[eeg]"
pip install -e ".[gnn]"

Running the Tests

The test suite uses only synthetic data — no NIfTI files or internet access required.

# Run all 387 tests
pytest

# With coverage report
pytest --cov=braphin --cov-report=term-missing

# Run a specific test module
pytest tests/test_transform.py -v

# Run tests matching a keyword
pytest -k "centroid" -v
Fast: the full suite runs in under 2 seconds. All tests use synthetic arrays generated with numpy.random — no real data files are needed.

All tests must pass before a pull request will be reviewed. New code must be accompanied by tests — aim for at least one happy-path test and one error-path test per public method.

Coding Style

RuleDetail
Line length100 characters (PEP 8)
Type hintsAll public function signatures (parameters and return type)
DocstringsAll public classes and methods (Google-style preferred)
No silent stubsIf a feature flag is not yet implemented, raise NotImplementedError — do not return input unchanged
ExceptionsRaise the most specific BRAPHINError subclass. Never raise a bare Exception
Bundle patternEach stage must accept and return a typed bundle dataclass. Never mutate input bundles
No real dataTests must use synthetic arrays. Real NIfTI files must never be committed

Submitting a Pull Request

  1. Create a feature branch from main: 
    git checkout -b feature/bandpass-filtering
  2. Make your changes, write tests, update CHANGELOG.md under [Unreleased].
  3. Ensure pytest passes with no failures or warnings.
  4. Push and open a pull request against main.
  5. Fill in the PR description: motivation, what changed, how it was tested.
Important: Pull requests that break existing tests or reduce coverage without justification will not be merged.

Commit Message Conventions

BRAPHIN uses Conventional Commits:

<type>(<scope>): <short summary>

[optional body]

[optional footer]
TypeWhen to use
featNew feature
fixBug fix
testAdding or correcting tests
docsDocumentation only
refactorCode change with no behaviour change
choreBuild scripts, CI, dependencies

Examples:

feat(denoise): add temporal bandpass filtering via scipy.signal
fix(tabular): skip BIDS TSV header row to avoid NaN confound rows
test(transform): add centroid space assertions for voxel-only atlases
docs: add HTML documentation site
